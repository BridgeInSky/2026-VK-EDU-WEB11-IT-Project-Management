# 2026-VK-EDU-WEB11-IT-Project-Management
Название проекта: SmartHome Hub (Центр умного дома)
=

Elevator Pitch
-
> Люди хотят "умный дом", но сталкиваются с выбором из 100 брендов, которые не дружат друг с другом, и не могут найти мастера, который согласится на мелкий заказ (просто поставить розетку). Магазины техники теряют покупателя на этапе "А кто мне это подключит?".
SmartHome Hub — это B2B2C-экосистема. Мы не берем комиссию с продаж устройств. Мы продаем подписку на "Мозги" (облачный контроллер, объединяющий все протоколы Zigbee/Matter/Wi-Fi) и продвижение для мастеров и магазинов. Пользователь покупает лампочку у любого партнера, активирует подписку за 5$ в месяц, и через нашу платформу заказывает мастера, который за 15 минут "оживит" систему. Магазины получают сервисный отдел без найма, мастера — поток заказов, а пользователи — реально работающий умный дом без головной боли

### 1. Роли и пользовательские истории

Выделено 4 ключевых роли платформы SmartHome Hub.

| ID | Роль | История (User Story) |
|----|------|----------------------|
| US1 | Покупатель (C) | Я хочу авторизоваться по номеру телефона в один клик, чтобы не запоминать пароль. |
| US2 | Покупатель (C) | Я хочу видеть в каталоге, какие устройства работают с экосистемой SmartHome Hub, чтобы понимать, что купленное устройство я смогу подключить к своему «мозгу» и управлять из одного приложения |
| US3 | Покупатель (C) | Я хочу нажать кнопку «Мастер рядом» на карточке любого товара, чтобы сразу вызвать установщика без поиска по сторонним сайтам. |
| US4 | Покупатель (C) | Я хочу создать сценарий «Если датчик движения сработал → включить свет в коридоре», даже если датчик и лампочка — разных брендов. |
| US5 | Покупатель (C) | Я хочу оплатить заказ на установку через приложение с защитой (деньги заморожены до завершения работы), чтобы не бояться мошенников. |
| US6 | Мастер | Я хочу создать профиль за 5 минут (ФИО, фото, зона выезда, сертификаты) и начать получать заказы без стартовых взносов. |
| US7 | Мастер | Я хочу получать push-уведомления о новых заявках в моём районе с ценой и расстоянием, чтобы не пропускать заказы. |
| US8 | Мастер | Я хочу выкупить «Приоритетный показ» на 7 дней за 300 ₽, чтобы моя анкета была выше в выдаче у клиента. |
| US9 | Магазин (B2B) | Я хочу загрузить прайс-лист из 200 товаров CSV-файлом и указать свою наценку, чтобы не вводить каждую позицию вручную. |
| US10 | Магазин (B2B) | Я хочу установить на свой сайт белый виджет «Вызвать мастера SHH», чтобы клиенты не возвращали товар из-за сложности установки. |
| US11 | Админ | Я хочу модернировать мастеров: проверять паспорт и ИНН до первого заказа, чтобы отсеять мошенников. |

### 3. MVP и MLP

#### MVP 
MVP запускается через 2 месяца разработки и проверяет гипотезы: "Люди готовы платить за удобный вызов мастера для умного дома".

| Компонент | Что входит в MVP |
|-----------|------------------|
| **Каталог устройств** | 50 товаров от 3 партнёров (ручной импорт). Фильтры: по цене, по протоколу (Zigbee/Wi-Fi). |
| **Авторизация** | По номеру телефона (SMS-код). Без соцсетей. |
| **Биржа мастеров** | Профиль мастера (имя, рейтинг, зона выезда, цена за час). Верификация через фото паспорта (ручная модерация). |
| **Заказ установки** | Кнопка «Мастер рядом» на карточке товара. Оплата 20% предоплаты через карту (заморозка до завершения). |
| **Уведомления** | Push/TG-уведомление мастеру о новом заказе. Кнопки «Принять» / «Отказаться». |
| **Отзывы** | 5-звёздочная система после завершения заказа. Обязательное поле «Что понравилось / не понравилось». |

#### MLP

Через 3-4 месяца после MVP добавляем функции, за которые пользователи полюбят продукт:

| Категория | Что добавляем |
|-----------|---------------|
| **Сценарии («мозги»)** | Конструктор «Если → То» для любых устройств разных брендов. Бесплатный тариф: 3 сценария, до 10 устройств. |
| **Платная подписка** | $5/мес: безлимит устройств, 50 сценариев, удалённый доступ, приоритетная техподдержка. |
| **Приоритетный показ мастеров** | Платное поднятие анкеты в топ (на 1, 7, 30 дней). |
| **Виджет для магазинов** | Белый iframe-виджет «Вызвать мастера SHH» для вставки на сайт партнёра. |
| **Масс-загрузка CSV** | Магазины загружают тысячи товаров за 1 файл. |

#### US3: Кнопка «Мастер рядом» на карточке товара

**Функциональные требования (ФТ)**

| № | Требование |
|---|-------------|
| ФТ1 | На каждой карточке товара ниже цены отображается блок с кнопкой «🔧 Вызвать мастера для установки». |
| ФТ2 | При нажатии открывается полупрозрачное модальное окно с полем «Ваш адрес» (автоподстановка из профиля, если есть). |
| ФТ3 | После ввода адреса система отображает до 3 ближайших мастеров (радиус 30 км) с указанием: имя, рейтинг (звёзды), цена за установку (фикс или час), время подачи. |
| ФТ4 | Рядом с каждым мастером — зелёная или жёлтая метка: «Свободен сегодня» / «Завтра». |
| ФТ5 | Пользователь выбирает мастера → нажимает «Выбрать и продолжить» → попадает на экран предоплаты (20% от суммы). |
| ФТ6 | До подтверждения заказа пользователь может сменить мастера (кнопка «Назад к списку»). |
| ФТ7 | Если в радиусе 30 км нет мастеров, показывается текст: «В вашем районе пока нет мастеров. Оставьте заявку — мы найдём специалиста». Появляется кнопка «Оставить заявку» (уходит в отдельный пул). |

**Нефункциональные требования (НФТ)**

| № | Требование |
|---|-------------|
| НФТ1 | Время отклика API при поиске мастеров (геозапрос по 10 000 мастеров) — не более 500 мс (95-й перцентиль). |
| НФТ2 | Кнопка и блок мастеров должны быть видны на экране без скролла (на десктопе и мобильных устройствах с разрешением 375px и выше). |
| НФТ3 | Интерфейс должен быть адаптивным: на мобильном кнопка занимает 100% ширины, на десктопе — 280px справа от описания товара. |
| НФТ4 | Цена за установку должна рассчитываться на основе средней стоимости за аналогичную работу в регионе за последние 30 дней (для прозрачности и защиты от завышения). |
| НФТ5 | Данные о мастерах кэшируются в Redis на 5 минут, чтобы не нагружать PostgreSQL при массовом просмотре товаров (1000+ RPS). |

#### US7: Push-уведомление мастеру о новом заказе

**Функциональные требования (ФТ)**

| № | Требование |
|---|-------------|
| ФТ1 | Мастер должен подписаться на уведомления в настройках профиля (Telegram, Web Push, либо SMS — резерв). По умолчанию — Telegram. |
| ФТ2 | При создании заказа пользователем система ищет подходящих мастеров (по геозоне, типу работ). Первым получает уведомление мастер с самым высоким рейтингом среди свободных. |
| ФТ3 | Уведомление содержит: тип устройства (умная розетка), адрес, расстояние до мастера (в км), цену за работу, кнопки «ПРИНЯТЬ» и «ОТКАЗАТЬСЯ». |
| ФТ4 | У мастера есть 10 минут на ответ. Если не ответил — заказ автоматически переходит следующему мастеру в очереди. |
| ФТ5 | Если мастер нажал «ПРИНЯТЬ», заказ переходит в статус «Мастер назначен». Пользователь видит контактный телефон мастера (через фейковый номер с переадресацией). |
| ФТ6 | Если мастер нажал «ОТКАЗАТЬСЯ», система фиксирует отказ и снижает его рейтинг «Принятых заявок» на 2% (влияет на приоритет в будущем). |
| ФТ7 | При технической ошибке доставки (не пришло уведомление) мастер может увидеть заказ в приложении в разделе «Доступные заказы» (обновляется каждые 30 секунд). |

**Нефункциональные требования (НФТ)**

| № | Требование |
|---|-------------|
| НФТ1 | Уведомление должно доставляться за < 1 секунды с момента создания заказа пользователем (измеряется от POST /orders до отправки в Telegram API). |
| НФТ2 | Доставка уведомлений дублируется: если Telegram API не ответил за 3 секунды, система отправляет Web Push. Если и он не доступен — SMS (резерв). |
| НФТ3 | Таймаут ответа мастера — 10 минут (+- 5 секунд на погрешность сети). По истечении заказ автоматически переводится к следующему мастеру. |
| НФТ4 | Логи всех уведомлений (когда отправлено, кому, доставлено ли, нажат ли ответ) хранятся 60 дней для разрешения споров (например, «Я не получал заказ!»). |
| НФТ5 | Система должна выдерживать пиковую нагрузку: 100 заказов в минуту → 100 уведомлений в минуту без задержек (горизонтальное масштабирование воркеров уведомлений). |

### 1. Доменные зоны (DDD)

Выделено 4 ограниченных контекста.

#### Домен 1: Управление клиентами (Client Management)

**Что делает:**  
Регистрация пользователей всех типов (покупатели, мастера, магазины), управление ролями, верификация через телефон/email.

**Глоссарий:**
- **Покупатель (Customer)** – конечный пользователь, который покупает устройства и заказывает мастеров.
- **Мастер (Master)** – специалист по установке и настройке умного дома.
- **Магазин (Store)** – B2B-партнёр, продающий устройства умного дома.
- **Верификация (Verification)** – подтверждение личности мастера (паспорт, ИНН).

#### Домен 2: Каталог и совместимость (Catalog & Compatibility)

**Что делает:**  
Управление устройствами, протоколами, проверка совместимости с платформой, фильтрация.

**Глоссарий:**
- **Устройство (Device)** – конкретный товар (умная розетка, лампочка, датчик).
- **Протокол (Protocol)** – язык общения устройства (Zigbee, Matter, Wi-Fi, MQTT).
- **Совместимость (Compatibility)** – возможность устройства работать с «мозгом» SHH.
- **Бренд (Brand)** – производитель устройства (Xiaomi, Aqara, Яндекс).

#### Домен 3: Заказы и услуги (Orders & Services)

**Что делает:**  
Создание заявок на установку, эскроу-платежи, статусы заказов, уведомления, отзывы.

**Глоссарий:**
- **Заказ на установку (Service Order)** – заявка от покупателя мастеру.
- **Эскроу (Escrow)** – заморозка предоплаты до завершения работ.
- **Статус заказа (Order Status)** – создан → мастер назначен → в работе → завершён.
- **Отзыв (Review)** – оценка мастера после завершения заказа.

#### Домен 4: Мастер-пространство (Master Space)

**Что делает:**  
Профили мастеров, геозоны, рейтинги, платное продвижение, KPI.

**Глоссарий:**
- **Геозона (GeoFence)** – район/город, где мастер готов работать.
- **Рейтинг (Rating)** – средняя оценка от покупателей (1–5 звёзд).
- **Приоритетный показ (Priority Listing)** – платное поднятие анкеты в топ.
- **KPI мастера** – процент принятых заявок, скорость ответа.

#### Домен 5: Подписка и биллинг (Subscription & Billing)

**Что делает:**  
Управление тарифами, подписками пользователей, платными услугами для мастеров.

**Глоссарий:**
- **Тариф (Plan)** – Free / Pro / Master.
- **Подписка (Subscription)** – регулярный платёж пользователя.
- **Платное продвижение (Promotion)** – разовая плата мастера за поднятие анкеты.
- **Комиссия (Commission)** – 5% с суммы заказа мастера (не с устройства).

### 2. BDD: Критический путь MVP

Критический путь – **покупатель заказывает установку устройства через биржу мастеров**.

#### Успешный сценарий

**Дано:**
- Покупатель Анна зарегистрирована на платформе.
- Мастер Дмитрий имеет подтверждённый профиль и находится в радиусе 5 км от Анны.
- Анна выбрала умную розетку в каталоге.

**Когда:**
Анна нажимает кнопку «Мастер рядом» → выбирает Дмитрия из списка → вносит предоплату 20%.

**Тогда:**
- Дмитрий получает push-уведомление и нажимает «Принять».
- Статус заказа становится «Мастер назначен».
- Анна видит контактный телефон мастера.
- Деньги предоплаты замораживаются на эскроу-счёте.

#### Сценарий отказа (мастер не ответил)

**Дано:**
- Те же Анна и Дмитрий.
- Дмитрий офлайн (не в сети).

**Когда:**
Анна создаёт заказ и вносит предоплату.

**Тогда:**
- Дмитрий не отвечает в течение 10 минут.
- Система переводит заказ в статус «Ищем другого мастера».
- Анна получает уведомление: «Мастер не ответил. Ищем замену».
- Дмитрию снижается KPI «процент принятых заявок» на 2%.

### 3. Технологический стек. Обоснование.

| Компонент | Выбор | Почему |
|-----------|-------|--------|
| **Frontend** | React + Vite + MUI | Быстрая разработка компонентов, PWA для push-уведомлений, адаптивность из коробки. |
| **Backend** | NestJS (Node.js) | TypeScript, модульность под DDD, встроенная поддержка WebSockets и MQTT. |
| **База данных** | PostgreSQL + PostGIS | Геопоиск мастеров (расстояние), ACID для платежей, надёжность. |
| **Очереди / Кэш** | Redis + Bull | Лёгкий, встроенный retry для уведомлений, быстрый кэш геопозиций. |
| **MQTT Брокер** | Mosquitto + aedes | Стандарт де-факто для IoT, лёгкий, хорошо масштабируется. |

### 2. Development Framework — Scrum

<u>Почему Scrum, а не Waterfall или Kanban:</u>

Жёсткий дедлайн (сдача проекта). Спринты по 2 недели дают понятные контрольные точки. Waterfall скрывает прогресс до конца, Kanban не имеет встроенного механизма «стоп, релиз».

Высокая неопределённость. Мы не знаем, как мастера и покупатели будут взаимодействовать с платформой. Scrum позволяет корректировать курс каждые 2 недели на основе реального фидбека.

Прозрачность для стейкхолдеров. В конце каждого спринта — демо (можно записать видео). Преподаватель видит прогресс, партнёры дают обратную связь.

Команда из 4 человек без опыта совместной работы. Scrum даёт чёткие роли (PO, Scrum Master, команда) и ежедневную синхронизацию (Daily).

### 3. Team Rituals (встречи)

| Ритуал | Цель | Частота и длительность |
|--------|------|------------------------|
| **Daily Standup** | Синхронизация, выявление блокеров. Каждый отвечает: что сделал вчера, что делаю сегодня, какие блокеры. | Каждый день (кроме понедельника — там Planning). 15 минут. |
| **Sprint Planning** | Выбрать истории на спринт из бэклога, оценить в story points (покер планирования), согласовать Sprint Goal. | Первый день спринта (понедельник). 1-1.5 часа. |
| **Sprint Review** | Показать работающий инкремент преподавателю/партнёрам. Собрать фидбек, который идёт в бэклог. | Последний день спринта (пятница). 30-45 минут. |
| **Sprint Retrospective** | Проанализировать процесс: что хорошо, что улучшить. Выбрать 1-2 улучшения на следующий спринт. | Сразу после Review. 30 минут. |
| **Backlog Refinement** | PO и разработчики уточняют детали будущих историй (acceptance criteria, технические риски). | Раз в неделю (среда). 45 минут. |
